SADAF Mobile API Documentation

This file is written for the mobile developer.
Goal: explain what each API is for, when it should be used in the mobile flow, what payload to send, what response to expect, and what non-obvious rules exist.

==================================================
1. Base Rules
==================================================

1.1. Base URL

Examples below use paths only:
- /users/login/
- /mobile/me
- /mobile/reservation-requests

In production you will call them with the real API domain.

1.2. Languages

Project supports 3 languages:
- en
- uz
- ru

Because Django uses i18n_patterns, language can be selected by URL prefix.
Examples:
- /mobile/about
- /uz/mobile/about
- /ru/mobile/about

Recommended:
- if app is currently Uzbek, use /uz/...
- if app is currently Russian, use /ru/...
- if app is currently English/default, /mobile/... usually works

Important:
- translated model fields such as article title, vacancy title, work title depend on active language
- translated validation messages and notification labels also depend on active language

1.3. Authentication

Authentication for protected APIs is JWT Bearer token.

Headers:
- Authorization: Bearer <access_token>

Public endpoints do not require token.
Protected endpoints do require token.

1.4. Date and Time Format

This project does NOT consistently use ISO date in mobile booking APIs.

Important formats:
- date: DD-MM-YYYY
- time: HH:MM
- datetime in many responses: DD-MM-YYYYTHH:MM

Examples:
- date: "22-03-2026"
- time: "14:30"

Do not send "2026-03-22" to reservation-related APIs unless that endpoint explicitly accepts it.
For booking flow, use DD-MM-YYYY.

1.5. General Success / Error Behavior

- Successful GET usually returns JSON object or JSON array
- List APIs may return paginated response if page is provided
- Validation errors usually return DRF error object
- Some older admin APIs may have inconsistent response shape, but mobile APIs are more consistent

2. Auth Flow
==================================================

2.1. Login

Endpoint:
- POST /users/login/

Purpose:
- authenticate existing user
- returns access and refresh tokens
- mobile must use returned access token for protected APIs

Typical payload:
{
  "username": "patient.portal.01",
  "password": "mock_patient_123"
}

Response notes:
- access token contains extra claims:
  - username
  - user_id
  - client_id
  - user_fullname
  - user_image

Use flow:
1. User enters login/password
2. Mobile calls /users/login/
3. Save access and refresh tokens
4. Use access token for /mobile/... protected endpoints
5. When access expires, refresh it

2.2. Token Refresh

Endpoint:
- POST /users/token/refresh

Payload:
{
  "refresh": "<refresh_token>"
}

Purpose:
- get new access token without forcing user to login again

8. Reservation Request Flow
==================================================

This is the core mobile booking flow.

Business meaning:
- mobile user does not directly create final reservation
- mobile creates ReservationRequest first
- clinic reviews it
- clinic approves or cancels it
- if approved, Reservation is created
- then client may confirm from mobile if frontend requires that step

8.1. Create Reservation Request

Endpoint:
- POST /mobile/reservation-requests

Auth:
- required

Payload:
{
  "flutter_reservation_id": "string",
  "doctor_id": 2,
  "reservation_work_id": 5,
  "note": "string",
  "date": "22-03-2026",
  "time": "14:00",
  "slot_minutes": 60
}

Field explanations:
- flutter_reservation_id
  - mobile-generated unique id from app side
  - purpose: app can match its local booking action with backend-created request
  - useful for retry logic, client-side tracking, analytics, or temporary local state
  - backend stores it in ReservationRequest.flutter_reservation_id
  - recommended: always send a unique string, for example UUID
- doctor_id
  - selected doctor primary key
- reservation_work_id
  - selected service/work primary key
- note
  - optional text from client
- date
  - booking date in DD-MM-YYYY
- time
  - slot start time in HH:MM
- slot_minutes
  - requested slot length in minutes
  - default is 60
  - minimum is normalized to 15

Important validation logic:
- user must be authenticated
- authenticated user must be linked to a Client
- doctor must be available
- selected time must not conflict with working hours/lunch/existing reservations
- selected time must not conflict with active pending request in that slot
- selected service must belong to doctor specialization if service has specialization restriction

What happens after success:
1. ReservationRequest is created with status=draft
2. doctor_name is copied and stored
3. notification is created for doctor side
4. mobile should show "waiting for clinic" state

8.2. My Reservation Request List

Endpoint:
- GET /mobile/reservation-requests

Auth:
- required

Purpose:
- show all current client request records

Useful response fields:
- id
- flutter_reservation_id
- doctor
- reservation_work
- doctor_name
- status
- status_code
- status_label
- ui_state
- next_action
- can_cancel
- can_confirm
- reservation_id
- note
- date
- time

Important meaning of helper fields:
- status_code
  - raw backend status key
  - examples: draft, approved, approved_by_patient, cancelled
- status_label
  - translated human-readable label
- ui_state
  - frontend-friendly short state:
  - pending
  - success
  - cancelled
- next_action
  - what UI usually should do next
  - wait_for_clinic
  - confirm_reservation
  - open_reservation
  - open_requests
- can_cancel
  - if true, show cancel action
- can_confirm
  - if true, show confirm action
- reservation_id
  - appears if clinic already created real Reservation

8.3. Reservation Request Detail

Endpoint:
- GET /mobile/reservation-requests/{id}

Auth:
- required

Purpose:
- detail screen for one request

8.4. Cancel Reservation Request

Endpoint:
- PATCH /mobile/reservation-requests/{id}/cancel

Auth:
- required

Purpose:
- client cancels own request

What happens:
- request status becomes cancelled_by_patient
- if Reservation already exists, that reservation is also marked cancelled
- doctor may receive notification

8.5. Confirm Reservation Request

Endpoint:
- PATCH /mobile/reservation-requests/{id}/confirm

Auth:
- required

Purpose:
- confirm request after clinic approved it

Important rule:
- this only works if request status is approved and real Reservation already exists

What happens:
- request status becomes approved_by_patient
- Reservation itself already exists before this call

12. Recommended Mobile Flow
==================================================

12.1. App Start

1. User logs in via /users/login/
2. Save access + refresh token
3. Call /mobile/dashboard
4. Optionally call /mobile/me if full profile screen opens

12.2. Booking Flow

1. Call /mobile/reservation/doctors
2. User selects doctor
3. Call /mobile/services?doctor_id={doctor_id}
4. User selects service
5. Call /mobile/reservation/doctors/{doctor_id}/slots?date=DD-MM-YYYY
6. User selects slot
7. Call POST /mobile/reservation-requests
8. Show waiting state from request status
9. Poll request list/detail or use notifications
10. If approved and can_confirm=true, call PATCH /mobile/reservation-requests/{id}/confirm
11. Open /mobile/reservations or /mobile/reservations/{reservation_id}

12.3. Loyalty / Referral Flow

1. Open /mobile/loyalty or /mobile/status
2. Show referral code and cashback
3. If user enters someone else’s code, call /mobile/referral/apply-code
4. Show updated cashback and referrals count

12.4. Notifications Flow

1. Home badge count -> /mobile/notifications/unread-count
2. Notification list -> /mobile/notifications
3. Open item -> inspect next_action + payload
4. Mark read -> /mobile/notifications/{id}/read

13. Important Non-Obvious Notes
==================================================

13.1. flutter_reservation_id

This is not the final reservation id.

It is a mobile-side tracking id for reservation request creation.
Backend stores it, but final confirmed reservation still gets its own reservation_id.

Recommended use:
- generate UUID on mobile before request creation
- save it locally
- if app wants to reconcile local pending state with server state, this field helps

13.2. doctor_id vs reservation_id vs reservation_request id

- doctor_id
  - selected doctor
- reservation request id
  - id of pending/approval request
- reservation_id
  - real confirmed reservation record

Do not mix them.

13.3. Booking date format

Reservation APIs expect:
- "22-03-2026"
not
- "2026-03-22"

If mobile sends ISO date here, validation may fail.

13.4. slot_minutes

Backend normalizes it.
- default = 60
- less than 15 becomes 15

Recommended:
- keep frontend values aligned with slot UI
- if your UI offers 60-minute slots, always send 60

13.5. Mobile public endpoints are read-only

- /mobile/articles
- /mobile/news
- /mobile/vacancies

These are for reading only.
Creation and admin management belong to CRM/admin APIs.

13.6. Loyalty is intentionally simple for now

Current tier/cashback logic is intentionally lightweight.
Business rules can change later.
Frontend should avoid hardcoding assumptions beyond response values.

15. Final Recommendation For Mobile Team
==================================================

For easiest integration, build screens in this order:
1. Login + token refresh
2. Dashboard
3. Doctor list + slot selection
4. Service selection
5. Create reservation request
6. My requests + My reservations
7. Notifications
8. Treatments
9. Loyalty / referral / status
10. Articles / news / vacancies

If anything in swagger and this document disagree, trust the current backend code first and clarify with backend before hardcoding assumptions.
