SADAF Mobile API Hujjati

Bu fayl mobile dasturchi uchun yozildi.
Maqsad: har bir API nima uchun kerakligini, mobile flow ichida qachon ishlatilishini, qanday payload yuborilishini, qanday response kelishini va ko'zga tashlanmaydigan muhim jihatlarni sodda tushuntirish.

==================================================
1. Asosiy Qoidalar
==================================================

1.1. Base URL

Quyidagi misollarda faqat path ko'rsatilgan:
- /users/login/
- /mobile/me
- /mobile/reservation-requests

Production holatda bular haqiqiy API domeni bilan birga chaqiriladi.

1.2. Tillar

Loyiha 3 tilni qo'llab-quvvatlaydi:
- en
- uz
- ru

Django `i18n_patterns` ishlatayotgani sababli til URL prefix orqali tanlanishi mumkin.
Misollar:
- /mobile/about
- /uz/mobile/about
- /ru/mobile/about

Tavsiya:
- ilova hozir o'zbek tilida bo'lsa, /uz/... ishlating
- ilova hozir rus tilida bo'lsa, /ru/... ishlating
- ilova ingliz/default holatda bo'lsa, odatda /mobile/... ham ishlaydi

Muhim:
- article title, vacancy title, work title kabi translate bo'ladigan fieldlar aktiv tilga bog'liq
- validation message va notification label lar ham aktiv tilga bog'liq

1.3. Authentication

Himoyalangan API lar JWT Bearer token orqali ishlaydi.

Header:
- Authorization: Bearer <access_token>

Public endpoint lar token talab qilmaydi.
Protected endpoint lar token talab qiladi.

1.4. Sana va Vaqt Formati

Bu project mobile booking API larida ISO sanani hamma joyda bir xil ishlatmaydi.

Muhim formatlar:
- date: DD-MM-YYYY
- time: HH:MM
- ko'p response lardagi datetime: DD-MM-YYYYTHH:MM

Misollar:
- date: "22-03-2026"
- time: "14:30"

Reservation bilan bog'liq API larga "2026-03-22" yubormang, agar o'sha endpoint ochiqchasiga shu formatni qabul qilishi ko'rsatilmagan bo'lsa.
Booking flow uchun DD-MM-YYYY ishlating.

1.5. Umumiy Success / Error Holati

- Successful GET odatda JSON object yoki JSON array qaytaradi
- page param berilsa list API lar pagination bilan kelishi mumkin
- Validation error lar odatda DRF error object ko'rinishida bo'ladi
- Ba'zi eski admin API lar response shape bo'yicha bir xil emas, lekin mobile API lar nisbatan tartibli

8. Reservation Request Flow
==================================================

Bu mobile booking flow ning yuragi hisoblanadi.

Biznes ma'nosi:
- mobile user to'g'ridan-to'g'ri final reservation yaratmaydi
- avval ReservationRequest yaratadi
- klinika uni ko'rib chiqadi
- klinika approve yoki cancel qiladi
- approve bo'lsa, real Reservation yaratiladi
- keyin frontend xohlasa client confirm qilishi mumkin

8.1. Reservation Request Yaratish

Endpoint:
- POST /mobile/reservation-requests

Auth:
- required

Payload:
{
  "flutter_reservation_id": "string",
  "doctor_id": 2,
  "reservation_work_id": 5,
  "note": "string",
  "date": "22-03-2026",
  "time": "14:00",
  "slot_minutes": 60
}

Field tushuntirishlari:
- flutter_reservation_id
  - mobile tomonidan generatsiya qilinadigan unique id
  - maqsad: app local booking action ni backenddagi request bilan bog'lay oladi
  - retry logika, client-side tracking, analytics yoki local pending state uchun foydali
  - backend uni ReservationRequest.flutter_reservation_id fieldida saqlaydi
  - tavsiya: har safar unique string yuboring, masalan UUID
- doctor_id
  - tanlangan doctor primary key si
- reservation_work_id
  - tanlangan service/work primary key si
- note
  - client yozadigan ixtiyoriy matn
- date
  - booking sanasi, DD-MM-YYYY formatida
- time
  - tanlangan slotning boshlanish vaqti, HH:MM formatida
- slot_minutes
  - so'ralayotgan slot uzunligi, minutlarda
  - default 60
  - minimum normalize qilinib 15 bo'ladi

Muhim validation logikasi:
- user authenticated bo'lishi kerak
- authenticated user Client ga bog'langan bo'lishi kerak
- doctor available bo'lishi kerak
- tanlangan vaqt working hours / lunch / existing reservations bilan to'qnashmasligi kerak
- tanlangan vaqt active pending request bilan to'qnashmasligi kerak
- tanlangan service doctor specialization ga mos bo'lishi kerak, agar service specialization restriction ga ega bo'lsa

Success bo'lgandan keyin nima bo'ladi:
1. ReservationRequest status=draft bilan yaratiladi
2. doctor_name nusxasi saqlanadi
3. doctor tarafga notification yaratiladi
4. mobile "wait_for_clinic" holatini ko'rsatishi kerak

12. Tavsiya Etilgan Mobile Flow
==================================================

12.1. App Start

1. User /users/login/ orqali login qiladi
2. Access + refresh token saqlanadi
3. /mobile/dashboard chaqiriladi
4. To'liq profile screen ochilsa /mobile/me chaqiriladi

12.2. Booking Flow

1. /mobile/reservation/doctors chaqiriladi
2. User doctor tanlaydi
3. /mobile/services?doctor_id={doctor_id} chaqiriladi
4. User service tanlaydi
5. /mobile/reservation/doctors/{doctor_id}/slots?date=DD-MM-YYYY chaqiriladi
6. User slot tanlaydi
7. POST /mobile/reservation-requests chaqiriladi
8. Request status asosida kutish holati ko'rsatiladi
9. Request list/detail poll qilinadi yoki notification lar ishlatiladi
10. Agar approved va can_confirm=true bo'lsa PATCH /mobile/reservation-requests/{id}/confirm chaqiriladi
11. /mobile/reservations yoki /mobile/reservations/{reservation_id} ochiladi

12.3. Loyalty / Referral Flow

1. /mobile/loyalty yoki /mobile/status ochiladi
2. Referral code va cashback ko'rsatiladi
3. User boshqa birovning referral kodini kiritsa /mobile/referral/apply-code chaqiriladi
4. Yangilangan cashback va referrals count ko'rsatiladi

12.4. Notifications Flow

1. Home badge count -> /mobile/notifications/unread-count
2. Notification list -> /mobile/notifications
3. Notification ochilganda next_action + payload ga qaraladi
4. Read qilish -> /mobile/notifications/{id}/read

13. Ko'zga Tashlanmaydigan Muhim Eslatmalar
==================================================

13.1. flutter_reservation_id

Bu final reservation id emas.

Bu mobile tarafdan reservation request yaratish vaqtida yuboriladigan tracking id.
Backend uni saqlaydi, lekin final confirmed reservation o'zining reservation_id si bilan yaratiladi.

Tavsiya:
- mobile request yaratishdan oldin UUID generatsiya qilsin
- local state da saqlab qo'ysin
- local pending state ni server record bilan solishtirishda foydali bo'ladi

13.2. doctor_id vs reservation_request id vs reservation_id

- doctor_id
  - tanlangan doctor
- reservation request id
  - pending / approval bosqichidagi request id
- reservation_id
  - real confirmed reservation id

Bularni aralashtirmang.

13.3. Booking date format

Reservation API lar odatda quyidagini kutadi:
- "22-03-2026"

quyidagini emas:
- "2026-03-22"

Mobile reservation API larga ISO sana yuborsa validation fail bo'lishi mumkin.

13.4. slot_minutes

Backend uni normalize qiladi.
- default = 60
- 15 dan kichik bo'lsa 15 ga ko'tariladi

Tavsiya:
- frontend qiymatlari slot UI bilan bir xil bo'lsin
- agar UI 60 minutlik slotlar bilan ishlasa, doim 60 yuboring

13.5. Public mobile endpoint lar read-only

- /mobile/articles
- /mobile/news
- /mobile/vacancies

Bu endpoint lar faqat o'qish uchun.
Yaratish va admin boshqaruvi CRM/admin tarafida bo'ladi.

13.6. Loyalty hozircha sodda

Hozirgi tier/cashback logikasi ataylab yengil yozilgan.
Business qoidalari keyin o'zgarishi mumkin.
Frontend response dagi qiymatga tayanishi kerak, ichki formula ni qattiq hardcode qilmasin.
